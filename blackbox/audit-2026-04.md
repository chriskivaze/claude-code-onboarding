1. Infrastructure & Queue Logic:

Configure Laravel to use RabbitMQ via AMQP.

Implement Durable Queues and Persistent Messages for all transaction-related jobs.

Set up a Dead Letter Exchange (DLX) named payments_failed_exchange to capture callbacks that exhaust all retry attempts.

Use Queue Routing (a new Laravel 13 feature: Queue::route()) to map MpesaCallbackJob to the high-priority RabbitMQ queue.

2. Multi-Tenant Reliability:

Idempotency: Use Redis (for speed) or the Database to store idempotency_keys for 24 hours. Every incoming STK/B2C request must verify this key before hitting the MpesaService.

Isolation: Use RabbitMQ Virtual Hosts (vhosts) or separate queues per internal service if high-volume isolation is required.

3. M-Pesa Service Layer (Laravel 13):

Use PHP 8.5 Attributes for DTOs (Data Transfer Objects) to handle M-Pesa's complex JSON responses.

Implement a PaymentGateway interface with methods for stkPush, b2c, b2b, and query.

Ensure the MpesaController validates Safaricom’s signature before pushing the raw payload into RabbitMQ.

4. The "Callback Consumer" Worker:

Create a specialized HandleServiceCallback job.

This job should attempt to POST the payment result to the internal service's webhook_url.

Use Exponential Backoff logic: If the internal service is down, RabbitMQ should delay the message and retry at 1m, 5m, 15m, and 1h intervals.

5. Monitoring Dashboard (Inertia + Vue):

Build a management UI that displays RabbitMQ queue health (Ready vs. Unacked messages).

Provide a "Replay" feature: Ability to grab a message from the Dead Letter Queue and re-queue it after fixing a bug.

Add a "Service Health" indicator based on the success rate of the last 100 webhooks delivered.

If a specific service's webhook failure rate exceeds 25% over 5 minutes, 'Open' the circuit for that service and stop attempting deliveries to them for 60 seconds to save resources, while logging the failures for later replay.
