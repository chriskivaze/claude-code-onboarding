# Deployment and CI/CD

> **When to use**: Setting up or modifying CI/CD pipelines, deploying to staging or production, or configuring GitHub Actions workflows
> **Time estimate**: 2–4 hours for initial CI/CD setup; 30 min per modification
> **Prerequisites**: GitHub repository, Docker configured, GCP project (for Cloud Run)

## Overview

End-to-end CI/CD workflow from code push to production deployment. Covers GitHub Actions pipeline design, Docker multi-stage builds, Artifact Registry push, Cloud Run staging deploy with health check, and production gate with manual approval. Uses `deployment-engineer` agent and `terraform-specialist` agent.

## Phases

### Phase 1 — Pipeline Design

**Trigger**: New service needs CI/CD, or existing pipeline needs modification
**Agent**: Dispatch `deployment-engineer`

**Pipeline structure** (from `deployment-engineer` agent capability):
```
Code Push / PR → Test → Build → Scan → Push → Deploy Staging → Health Check → [Manual Gate] → Deploy Production
```

**Branch strategy**:
| Branch | Pipeline Trigger | Deploys To |
|--------|-----------------|-----------|
| `feature/*` | PR opened/updated | Staging (optional) |
| `develop` | Merge | Staging (automatic) |
| `main` / `release/*` | Merge | Production (manual gate) |

### Phase 2 — GitHub Actions Workflow

**Generated by `deployment-engineer` agent**:

```yaml
# .github/workflows/deploy.yml
name: Deploy

on:
  push:
    branches: [develop, main]
  pull_request:
    branches: [develop]

env:
  PROJECT_ID: ${{ vars.GCP_PROJECT_ID }}
  REGION: us-central1
  SERVICE_NAME: my-service
  REGISTRY: ${{ vars.GCP_REGION }}-docker.pkg.dev

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run tests
        run: |
          # Java: ./mvnw test
          # NestJS: npm ci && npx vitest run
          # Python: uv run pytest
          # Flutter: flutter test

  build:
    needs: test
    runs-on: ubuntu-latest
    permissions:
      id-token: write   # Workload Identity Federation
      contents: read
    steps:
      - uses: actions/checkout@v4

      - name: Authenticate to GCP
        uses: google-github-actions/auth@v2
        with:
          workload_identity_provider: ${{ vars.WIF_PROVIDER }}
          service_account: ${{ vars.WIF_SERVICE_ACCOUNT }}

      - name: Build Docker image
        run: |
          docker build -t $REGISTRY/$PROJECT_ID/$SERVICE_NAME:${{ github.sha }} .

      - name: Scan image (Trivy)
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: $REGISTRY/$PROJECT_ID/$SERVICE_NAME:${{ github.sha }}
          exit-code: 1
          severity: CRITICAL,HIGH

      - name: Push to Artifact Registry
        run: |
          gcloud auth configure-docker $REGISTRY
          docker push $REGISTRY/$PROJECT_ID/$SERVICE_NAME:${{ github.sha }}

  deploy-staging:
    needs: build
    if: github.ref == 'refs/heads/develop'
    runs-on: ubuntu-latest
    environment: staging
    steps:
      - name: Deploy to Cloud Run (staging)
        run: |
          gcloud run deploy $SERVICE_NAME-staging \
            --image $REGISTRY/$PROJECT_ID/$SERVICE_NAME:${{ github.sha }} \
            --region $REGION \
            --platform managed \
            --no-allow-unauthenticated

      - name: Health check
        run: |
          URL=$(gcloud run services describe $SERVICE_NAME-staging --region $REGION --format 'value(status.url)')
          curl -f "$URL/health" || exit 1

  deploy-production:
    needs: deploy-staging
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    environment: production   # Requires manual approval in GitHub
    steps:
      - name: Deploy to Cloud Run (production)
        run: |
          gcloud run deploy $SERVICE_NAME \
            --image $REGISTRY/$PROJECT_ID/$SERVICE_NAME:${{ github.sha }} \
            --region $REGION \
            --platform managed \
            --no-allow-unauthenticated \
            --traffic 10        # Staged rollout — start at 10%
```

### Phase 3 — Docker Multi-Stage Build

**Generated by `deployment-engineer` agent**:

```dockerfile
# Dockerfile — NestJS example
FROM node:22-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM node:22-alpine AS production
WORKDIR /app
# Only copy production artifacts
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
COPY package*.json ./

# Non-root user (security)
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser

EXPOSE 3000
CMD ["node", "dist/main.js"]
```

**Multi-stage principles**:
- Builder stage: has dev dependencies, build tools
- Production stage: only runtime artifacts, no dev dependencies
- Non-root user: security hardening
- `.dockerignore`: exclude node_modules, .git, test files, .env

### Phase 4 — Cloud Run Configuration

**Agent**: `terraform-specialist` for infrastructure, `deployment-engineer` for deployment config

**Cloud Run service config** (from `terraform-specialist` agent):
```hcl
# terraform/modules/cloud-run/main.tf
resource "google_cloud_run_v2_service" "service" {
  name     = var.service_name
  location = var.region

  template {
    containers {
      image = var.image_url

      resources {
        limits = {
          cpu    = "1"
          memory = "512Mi"
        }
      }

      env {
        name  = "DATABASE_URL"
        value_source {
          secret_key_ref {
            secret  = google_secret_manager_secret.db_url.secret_id
            version = "latest"
          }
        }
      }
    }

    scaling {
      min_instance_count = 1
      max_instance_count = 10
    }
  }

  traffic {
    type    = "TRAFFIC_TARGET_ALLOCATION_TYPE_LATEST"
    percent = 100
  }
}
```

### Phase 5 — Workload Identity Federation (No Service Account Keys)

**Agent**: `terraform-specialist` + `security-reviewer`

```hcl
# terraform/modules/wif/main.tf — No long-lived keys
resource "google_iam_workload_identity_pool" "github_pool" {
  workload_identity_pool_id = "github-pool"
  display_name              = "GitHub Actions Pool"
}

resource "google_iam_workload_identity_pool_provider" "github_provider" {
  workload_identity_pool_id          = google_iam_workload_identity_pool.github_pool.workload_identity_pool_id
  workload_identity_pool_provider_id = "github-provider"

  oidc {
    issuer_uri = "https://token.actions.githubusercontent.com"
  }

  attribute_mapping = {
    "google.subject"       = "assertion.sub"
    "attribute.repository" = "assertion.repository"
  }
}
```

**Why**: No service account JSON keys stored in GitHub Secrets — short-lived tokens issued per job via OIDC.

### Phase 6 — Traffic Splitting (Progressive Delivery)

```bash
# Deploy new version, send 10% traffic
gcloud run services update-traffic my-service \
  --to-revisions=my-service-00100-abc=10,LATEST=10

# Monitor error rate — if healthy, increase
gcloud run services update-traffic my-service \
  --to-revisions=LATEST=100
```

**Rollback** (immediate):
```bash
gcloud run services update-traffic my-service \
  --to-revisions=my-service-00099-xyz=100  # Previous stable revision
```

## Common Pitfalls

- **Service account JSON keys in GitHub Secrets** — use Workload Identity Federation instead; keys are a credential leak risk
- **No image scanning** — Trivy catches known CVEs in base images; add before push, not after deploy
- **Skipping staging** — staging must match production config; different env vars = different behavior
- **Manual deployment to production** — require `environment: production` with GitHub environment protection rules (approval gate)
- **No rollback plan** — always know the previous stable revision before deploying; Cloud Run traffic splitting enables instant rollback
